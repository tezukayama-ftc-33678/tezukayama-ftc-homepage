# Tezukayama-Rise サイト運用ガイド（非技術者向け）

このリポジトリはシンプルな静的サイトです。非技術者でも簡単に本文を編集できるよう、ページ内の文言は `content.json` にまとめています。以下は編集・プレビュー・デプロイの手順と LLM 活用のコツです。

**重要なファイル**
- `index.html` — サイトのテンプレート（構造・デザイン）
- `styles.css` — スタイル
- `script.js` — 画面操作と `content.json` の読み込み
- `content.json` — ページ内の文言を保持。ここを編集してください。

編集の流れ（簡単）

1. VS Code でリポジトリを開く。
2. `content.json` を編集する（日本語文を直接修正）。
3. ローカルサーバーでプレビューする（次節参照）。

ローカルでプレビューする方法

- 推奨: VS Code の `Live Server` 拡張を使う（ファイル保存で自動リロード）。
- 代替（Pythonがあれば）:

```powershell
# リポジトリのルートで実行
python -m http.server 8000
# ブラウザで http://localhost:8000 を開く
```

注: `file://` で直接開くと `fetch('content.json')` が動作しないことがあります。必ずローカルサーバーでプレビューしてください。

安全に編集するコツ（非技術者向け）

- `content.json` は厳密な JSON 形式です。カンマや引用符を間違えないでください。
- 改行を入れたいときは `<br>` を使えます（`heroDescription` など）。ただし `<script>` 等のタグは使わないでください。
- 編集前にファイルを保存し、ブラウザでリロードして反映を確認してください。

LLM を使った編集テンプレート（そのまま使えます）

非技術者が LLM に編集を頼むときの例プロンプト（日本語）:

```
以下のJSONキーに対応する短い日本語文（表示用）を返してください。出力は有効なJSONのみで返してください。
キー一覧: teamNumber, teamName, heroTitle, heroSubtitle, heroDescription, aboutTitle, aboutMission.title, aboutMission.text, aboutValues.title, aboutValues.text, aboutGoals.title, aboutGoals.text, outreach.cards[0..2].title, outreach.cards[0..2].text, contact.email.title, contact.email.text, contact.social.title, contact.social.text, contact.team.title, contact.team.text, footer.teamName, footer.copy

トーン: 明るく、教育的、親しみやすい
```

LLM によって生成されたJSONをそのまま `content.json` に貼れば、サイトに反映できます。必要ならこちらで差分編集も行います。

デプロイ方法（簡単）

- GitHub Pages: `main` ブランチに push すれば公開できます（リポジトリ設定で Pages を有効に）。

運用上の注意

- HTML や CSS の構造変更は開発者に依頼してください。非技術者の方は `content.json` の編集に限定してください。
- 外部サービスやフォームの追加などはセキュリティ確認のため開発者と相談してください。

サポート

編集したい箇所や文言の変更案があれば教えてください。LLM用のプロンプト作成や `content.json` の更新をこちらで代行できます。

**AI（LLM）にファイルを渡して編集を依頼する方法**

非技術者でも安全かつ効率的に編集するために、LLM（例: ChatGPT）にプロジェクト内のファイルを渡して自動で編集させる手順を簡潔にまとめます。これは『LLMに編集提案を作ってもらい、出力を自分で確認して反映する』ための安全なワークフローです。

手順（安全に進めるための推奨フロー）

1. 変更前にリポジトリをバックアップ（ローカルでコピー、または新しいブランチを作成）。
2. LLMに渡すファイルを決める（通常は `content.json` のみで十分です。構造変更が必要な場合は `index.html` の該当部だけを渡してください）。
3. LLMに対して「出力を厳密なJSONだけで返す」ように指示する。
4. 生成されたJSONやHTMLは必ず自分で検証する（JSON整合性、HTML/スクリプトの不正なタグが無いか）。
5. 問題なければ差分をリポジトリにコミットして push してください。

安全上の注意

- LLMにリポジトリ全体や秘密情報（APIキー、パスワード、個人情報）を渡さないでください。
- LLM出力は必ず人間が確認してください。自動でそのまま公開するのは避けてください。

例：渡すファイルと簡単なプロンプト（日本語）

```
ファイル: content.json

依頼文:
"以下の `content.json` を受け取ります。トーンは「明るく、教育的、親しみやすい」です。アウトリーチに新しいカードを1件追加し、既存のカード文言を短く整えてください。出力は更新後の有効なJSONのみで返してください。"
```

LLMに渡すときのヒント

- 変更は小さく分ける（1回に多くの変更を頼まない）と検証が楽です。
- LLMには「必ずJSONのみを返す」「コードブロック以外の説明を返さない」と明記してください。

この方法で、非技術者の方がLLMを安全に活用して `content.json`（や限定したHTMLスニペット）を編集できます。必要であれば、私がLLM向けの具体的プロンプトを作成して代行生成します。

**ローカル編集UI と 検証スクリプト**

簡易的な編集UIと検証スクリプトを用意しました（ローカルで使うことを前提）。

- 編集UI: `editor.html` をローカルサーバーで開くと、`content.json` の読み込み・編集・カードの追加/削除・ダウンロードができます。ブラウザの「ファイルを直接開く (file://)」ではfetchがブロックされるため、必ずローカルサーバーで実行してください。

- 編集UI はファイルアップロードにも対応しました: `editor.html` のファイル入力からローカルの JSON を読み込んで編集できます（上書きではなく読み込み・編集後にダウンロードして差し替えるワークフローです）。

	起動例:

	起動例:
	```powershell
	python -m http.server 8000
	# ブラウザで http://localhost:8000/editor.html を開く
	```

- 検証スクリプト: `scripts/verify.py` を実行すると、`content.json` の構文や簡易セキュリティチェック（`<script>` や `javascript:` の検出）、および必須ファイルの存在チェックを行います。

	実行例:

**新しいページを追加したい場合**

- 手順（簡易）:
	1. `page-template.html` をコピーして、新しいファイル名（例: `events.html`）を付けます。
	2. コピーしたファイル内の本文を編集します。
	3. ナビゲーションにリンクを追加するには、`index.html` 内の `.nav-menu` に `<li><a href="events.html">イベント</a></li>` のように追加してください。
	4. ローカルサーバーで動作確認してからリポジトリにコミット・push してください。

- メモ: ページ内で同じ `styles.css` と `script.js` を使うため、デザインや動作は統一されます。ページ固有のスクリプトやスタイルが必要な場合は、追加のCSS/JSファイルを作成して参照してください（開発者への相談推奨）。

**CSS の色を変えたい場合（簡単）**

- `styles.css` の先頭近くに `:root` ブロックがあります。ここでカラーパレットを変えるだけでサイト全体の色を変更できます。

	例（該当ファイル）: [styles.css](styles.css#L1-L40)

- よく使う変数の例:
	- `--primary-color` — ボタンやアクセントの基本色
	- `--secondary-color` — 補助的なアクセント
	- `--accent-color` — 強調に使う色
	- `--dark-bg` / `--light-bg` — 背景色

- 変更手順:
	1. `styles.css` を開く。
	2. `:root` の該当変数値を好きな16進カラー（例: `#1e90ff`）に書き換える。
	3. 保存してローカルサーバーで確認する。

- カラーの組み合わせが心配なら、次のキーワードで検索して使えるツールを試してください: "カラー パレット ジェネレーター"（例: Coolors, Adobe Color）。

必要であれば、代表的な配色候補を 3 パターン用意します。どのトーンが良いか教えてください（例: 明るい/落ち着いた/コントラスト高め）。

**カード（アウトリーチ / about）の増減 - 具体手順**

1. `content.json` を開く。
2. セクションごとに以下の配列を編集してください。
	 - アウトリーチ: `outreach.cards` 配列（各要素は `{ "icon": "📚", "title": "...", "text": "..." }`）
	 - About: `about.cards` 配列（各要素は `{ "icon": "🤖", "title": "...", "text": "..." }`）
3. カードを追加する場合は配列に新しいオブジェクトを push、削除する場合は該当オブジェクトを配列から削除します。
4. 保存してブラウザをリロードしてください（ローカルサーバーが必要）。サイトは配列の長さに合わせてカードを自動で表示します。

例: `outreach.cards` に新しいカードを追加する例

```json
{
	"icon": "🔬",
	"title": "出張講義",
	"text": "学校やイベントでの出張講義を受け付けています。"
}
```

注意:
- 追加する `text` には HTML の簡単なタグ（例: `<br>`）を使えますが、`<script>` のような実行可能なタグは絶対に入れないでください。
- JSON の文法（カンマ、引用符）を崩すとページが表示されなくなります。VS Code の JSON 検証を利用してください。

ナビへの自動リンク挿入制御

追加した任意セクションは自動的にナビゲーションにリンクが生成されます。生成されるリンクの挿入位置は `content.json` 側で制御できます。

セクションの例（`navPosition` または `navIndex` を使用）:

```json
{
	"key": "events",
	"title": "イベント情報",
	"navPosition": "start",    // 'start'|'end'（末尾）または省略
	"navIndex": 2,              // 数値を指定するとそのインデックスに挿入（0ベース）。優先度は navIndex -> navPosition
	"cards": [ ... ]
}
```

使い分けのポイント:
- `navPosition: "start"` — ナビの先頭に追加
- `navPosition: "end"` または未指定 — ナビの末尾に追加（デフォルト）
- `navIndex: <number>` — 0ベースの位置を指定（他の自動生成リンクを含む現在のナビ要素に対するインデックス）

注意: `editor.html` の「追加セクション」ではナビ挿入位置を選べます（`none`/`start`/`end`/`index`）。`index` の場合は数値を入力してください。

画像の扱いについて

- カードやセクションヘッダーに画像を追加できます。編集UI (`editor.html`) の各カードに「画像URL」フィールドとファイルアップロードがあります。アップロードすると画像は data URL として JSON に埋め込まれます（便利だがファイルサイズに注意）。
- 画像は `card.imageUrl` または `section.imageUrl` に指定します（URL か data URL）。
- 公開や配布時は、可能なら画像は静的ファイルとして `images/` に置き、`imageUrl` に相対パスを指定することを推奨します（data URL は大きくなりがちです）。
- アップロード画像はクライアントサイドでエンコードされます。外部サイトにアップロードしないため簡単ですが、サイズに注意してください。

サンプル: `images/` に画像を置く方法

1. 画像ファイルをリポジトリ内の `images/` フォルダに置きます（例: `images/sample-event.svg`）。
2. `content.json` の該当カードに相対パスを指定します：

```json
{
	"title": "春のワークショップ",
	"text": "次回は○月○日に開催予定です。",
	"imageUrl": "images/sample-event.svg"
}
```

3. ブラウザをリロードすると画像が表示されます。

注意: 画像をGitで管理する場合、ファイルサイズに注意してください（大きすぎる画像はリポジトリ肥大の原因になります）。必要なら `images/README.md` に管理ルールを追加してください。
